LLM 应用开发实践：RAG 系统的构建与优化

DJ· 2026-02-28· 12 分钟阅读

RAG（Retrieval-Augmented Generation，检索增强生成）是目前 LLM 应用落地最成熟的方案之一。本文基于我构建 ChatContext 的实战经验，从向量数据库选型到 Prompt 工程，系统梳理 RAG 的核心环节。

⚠️ 本文假设你对 LLM 基本概念（Token、Embedding、Prompt）有一定了解。代码示例使用 Python + LangChain。

为什么需要 RAG

LLM 有两个固有缺陷：知识截止日期和上下文窗口限制。当你需要模型回答关于你公司内部文档、最新数据或私有知识库的问题时，微调成本太高，而 RAG 是更轻量的解法。

RAG 的核心思路是：先检索，再生成。在提问时，先从知识库中检索相关片段，将其塞入 Prompt，再让模型基于这些上下文生成答案。

第一步：文档处理与分块

文档处理是整个流程中最容易被忽视、也最影响效果的环节。分块策略直接决定检索质量：

  • 块太大：语义噪声多，检索精度下降，同时占用更多上下文窗口
  • 块太小：缺少上下文，模型无法生成完整答案

我的经验是：对于结构化文档（如技术手册），按标题层级分块效果最好；对于非结构化文本，使用 RecursiveCharacterTextSplitter，设置 chunk_size=512、overlap=64 是一个不错的起点。

第二步：向量数据库选型

🏠 本地/自托管

  • Chroma — 开发环境首选，零配置
  • Qdrant — 生产级，性能优秀
  • pgvector — 已有 PostgreSQL 首选

☁️ 云服务

  • Pinecone — 最成熟，文档完善
  • Weaviate Cloud — 支持混合搜索
  • Supabase Vector — 全栈友好

我在 ChatContext 中使用 pgvector，因为项目本身已经使用 PostgreSQL，统一技术栈降低了运维复杂度。

第三步：提升检索质量

朴素的向量相似度搜索效果有限，以下几个优化对实际效果提升明显：

混合搜索（Hybrid Search）

将向量搜索（语义相似）和关键词搜索（BM25）的结果融合，用 RRF（Reciprocal Rank Fusion）重排序。对专业术语、产品名称等查询效果显著提升。

查询改写（Query Rewriting）

用 LLM 将用户的口语化提问改写为更适合检索的标准表达，或扩展为多个子查询，再合并结果。

父子分块（Parent-Child Chunking）

检索时用小块（高精度），生成时返回其父块（完整上下文）。这是目前效果最好的分块策略之一。

第四步：Prompt 工程

一个好的 RAG Prompt 应该包含：清晰的角色定义、上下文注入格式、引用要求、拒绝回答的条件（当检索内容不足以支撑答案时，让模型明说，而不是编造）。

关键原则：告诉模型它不知道什么，比告诉它知道什么更重要。

常见问题 & 解决方案

  • 幻觉问题：要求模型每句话都引用来源，无法引用则不输出
  • 答案不完整：增加检索 Top-K 数量，或使用父子分块扩大上下文
  • 延迟高：异步预处理文档；对高频查询缓存 Embedding 结果

总结

RAG 并不是银弹，它本质上是在检索质量和生成质量之间找平衡。建议从最简单的方案开始，先跑通 pipeline，再逐步加入混合搜索、重排序等优化，每次只改一个变量，这样才能知道每个优化真正带来了多少收益。
