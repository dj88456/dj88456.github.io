Azure RBAC and Cloud Governance: A Practical Framework

Alex Dai· 2026-01-20· 8 min read

As organizations expand their Azure footprint, access control becomes one of the most critical — and most frequently mishandled — aspects of cloud governance. Too permissive and you have a security incident waiting to happen. Too restrictive and your engineers can't do their jobs. This article describes the RBAC model and governance framework we implemented at WCB-Alberta.

💡 This article pairs well with the Azure Landing Zone article. The governance structures described here assume a hub-and-spoke subscription model with management groups already in place.

Understanding Azure RBAC Fundamentals

Azure Role-Based Access Control (RBAC) is an authorization system built on Azure Resource Manager. Access is granted by assigning a role (a collection of permissions) to a security principal (user, group, service principal, or managed identity) at a specific scope (management group, subscription, resource group, or resource).

The key principle is inheritance: permissions assigned at a higher scope cascade down. A Contributor role at the subscription level grants Contributor access to every resource group and resource within that subscription. This makes scope selection critically important — assign at the lowest scope that satisfies the use case.

Built-in Roles vs. Custom Roles

Azure provides over 100 built-in roles. For most use cases, the following five cover the vast majority of scenarios:

RoleWhat It Can DoTypical Assignment
OwnerFull access including role assignmentPlatform team on platform subscriptions only
ContributorCreate/manage all resources, no role assignmentApplication teams on their own resource groups
ReaderView all resources, no modificationsAuditors, security team, senior management
User Access AdministratorManage access onlyDelegated access managers (use sparingly)
Network ContributorManage networking resourcesNetwork team on spoke VNets

Custom roles are powerful but come with maintenance overhead. We created custom roles only when built-ins were either too permissive (granted more than needed) or too restrictive (required combining permissions from multiple built-ins). Our two most useful custom roles:

  • VM Operator — Can start/stop/restart VMs but cannot create, delete, or modify them. For helpdesk staff who need operational access without deployment rights.
  • Monitoring Reader Plus — Reader plus the ability to query Log Analytics workspaces. For the security team who needed to run KQL queries without broader Contributor access.

Structuring Access with Groups

A common mistake is assigning roles directly to individual users. This creates a sprawl of individual role assignments that becomes impossible to audit and maintain. Always assign roles to Azure AD groups, never to individuals.

Our group naming convention: [Department]-[Environment]-[Role]. For example:

  • IT-Prod-Contributor — IT team with Contributor on production subscriptions
  • AppTeamA-Dev-Contributor — Application Team A with Contributor on their dev resource group
  • Security-All-Reader — Security team with Reader across all subscriptions

Group membership is managed in Azure AD (now Entra ID), which means you can use Privileged Identity Management (PIM) to make sensitive group memberships time-bound and require approval.

Privileged Identity Management (PIM)

PIM is one of the highest-value governance tools in the Microsoft 365 / Azure ecosystem, and it's frequently overlooked. Rather than granting permanent Owner or Global Administrator access, PIM allows you to make these roles eligible — users can activate them on-demand for a defined period (e.g., 8 hours) with optional MFA re-authentication and justification.

We configured PIM for all roles with */write permissions at the subscription scope or above. Key settings we use:

  • Maximum activation duration: 8 hours for Contributor, 4 hours for Owner
  • Require MFA on activation: enabled for all elevated roles
  • Require justification: enabled — engineers must provide a ticket number
  • Notify on activation: alerts to the security team distribution list

Azure Policy for Guardrails

RBAC controls who can do what. Azure Policy controls what can be done, regardless of who is asking. The two are complementary — RBAC alone is insufficient for governance because Contributor-level access is very broad.

Our policy framework is organized into three tiers:

Tier 1 — Deny Policies (non-negotiable)

  • Deny resources outside Canada Central and Canada East
  • Deny public IP addresses on VMs in production
  • Deny storage accounts with public blob access
  • Deny SQL servers without Azure AD authentication enforced

Tier 2 — Audit Policies (visibility)

  • Audit VMs not using managed disks
  • Audit Key Vaults without soft delete enabled
  • Audit resources missing required tags (CostCenter, Owner, Environment)

Tier 3 — DeployIfNotExists Policies (auto-remediation)

  • Deploy diagnostic settings to Log Analytics if missing
  • Enable Microsoft Defender for Cloud on new subscriptions
  • Deploy network watcher when a VNet is created

Governance Review Cadence

A governance framework is only as good as the review process behind it. We run three recurring reviews:

  • Monthly: Access review via Entra ID Access Reviews — all Owners and privileged group members must re-certify their access
  • Quarterly: Policy compliance dashboard review — identify and remediate persistent violations
  • Annually: Full RBAC audit — remove stale role assignments, review custom role definitions, assess whether any roles should be elevated to PIM-only

Common Pitfalls to Avoid

  • Subscription-level Owner for application teams: Unnecessary and dangerous. Application teams should have Contributor scoped to their resource groups, not the whole subscription.
  • Service principals with Contributor at subscription scope: Automation accounts often get over-privileged because it's easier. Scope them to the specific resource groups they actually need.
  • Skipping PIM for "temporary" elevated access: Temporary access has a way of becoming permanent. PIM's time-bounded activations enforce the "temporary" part automatically.
  • Not testing Deny policies before assigning: Always test with Audit mode first. We had one Deny policy that inadvertently blocked a legitimate diagnostic setting because the policy condition wasn't precise enough.

Final Thoughts

Governance isn't a one-time project — it's an ongoing practice. The frameworks described here took several months to fully implement and refine. Start with the fundamentals: group-based RBAC, PIM for elevated roles, and a small set of Deny policies for your absolute non-negotiables. Then build out from there as your cloud footprint grows and your team's comfort with the tooling increases.

The goal is to reach a state where your engineers have the access they need to move fast, and your security team can verify compliance without manual work.
